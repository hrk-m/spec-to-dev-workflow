# ランタイム証拠ベースのデバッグ（プロンプト）

コードだけで原因を断定せず、**仮説 → 計測 → 再現 → ログ分析 → 修正 → 検証 → 掃除** の順で進める。Cursor のデバッグモードでセッション用の **ingest URL・ログファイルパス・Session ID** が渡される場合は、それらを必ず使う（スキル本文に固定値は書かない）。

## いつ使うか

- 画面や API の挙動が **再現したりしなかったり** する
- エラーメッセージが汎用的で **スタックや原因が隠れている**（例: `catch` で一律メッセージに置換）
- 本番に近い環境でしか起きない、タイミング依存がありそう

## 禁止事項

- ログなしで「たぶんここ」と **100% 確信して修正** しない
- **setTimeout / sleep を「修正」として** 入れない（待ちは調査用のみ、本修正はイベント・正しいライフサイクルで）
- **秘密情報**（トークン、パスワード、個人を特定できる情報）をログに含めない

---

## ステップ1: 仮説を立てる（3〜5個以上）

失敗経路ごとに **検証可能な仮説** を列挙する。例:

- どの **API / DB / 外部I/O** が失敗しているか（一覧取得か、後続のまとめ取得か、など）
- **バリデーション**（Zod 等）が特定の入力でだけ落ちるか
- **データ形状**（枝番付き ID を枝番なし前提の API に渡している等）のミスマッチ
- **競合・二重実行**（Strict Mode、Suspense、複数リクエスト）

各仮説に **ID**（`H1`, `H2`, …）を付け、後のログの `hypothesisId` と対応させる。

---

## ステップ2: コードを読み、計測点を決める

1. エラーメッセージ・フォールバック UI の **表示元**（コンポーネント・Server Action・Route）を特定する。
2. 失敗が **どの async 境界** で起きうるかを列挙する。
3. 仮説を **一括で潰せる最小のログ数**（目安 2〜10 個）に収める。

**ステージ変数**（例: `parse` → `listX` → `enrichY`）を 1 本持ち、`catch` で **最後に成功したステージ** を出すと、どの await が落ちたかがログだけで分かる。

---

## ステップ3: 計測（インストルメンテーション）を入れる

### 共通ルール

- 各ログに **`hypothesisId`** と、分かるなら **`runId`**（初回 / `post-fix` 等）を付ける。
- エージェント向けの折りたたみ: `// #region agent log` … `// #endregion`（言語に合わせる）。
- **ブラウザ用** と **サーバー用** を混同しない。Server Action / `"use server"` では **サーバー** が実行するため、送信先がローカル ingest の場合は **開発サーバーから `127.0.0.1` に届くか** を意識する（届かない環境ではファイル追記ログ等の代替を検討）。

### JavaScript / TypeScript（HTTP ingest がある場合）

システムから **Server endpoint** と **Session ID** が渡されたとき、指示どおり **POST** する（`fetch` の URL・ヘッダは **そのセッションの値** を使う。スキルにハードコードしない）。

ペイロード例の構造:

```json
{
  "sessionId": "<システムから渡された値>",
  "runId": "pre-fix",
  "hypothesisId": "H2",
  "location": "path/to/file.ts:関数名",
  "message": "短い説明",
  "data": { "stage": "listEntries", "count": 42 },
  "timestamp": 1730000000000
}
```

### ログファイルのみ指定される場合

**NDJSON**（1 行 1 JSON）で **指定パスに追記** する方法を使う。

### 再現前のログクリア

**再現依頼の直前** に、指定の **自分のセッション用ログファイルだけ** を空にする。ツールで削除できるなら削除し、**他セッションのログファイルは触らない**。シェルの `rm` 等はルールで禁じられている場合があるので、指示に従う。

---

## ステップ4: ユーザーに再現してもらう

1. 具体的な操作手順（URL、クリック、リロード回数など）を書く。
2. 必要なら **開発サーバー再起動が必要か** を一言添える。
3. インターフェース依存の表現は避け、完了用の一文はプロダクトの指示に合わせる（例: 「完了したら Proceed を押す」等）。

---

## ステップ5: ログで仮説を判定する

ログ各行について **CONFIRMED / REJECTED / INCONCLUSIVE** を付け、**根拠になるフィールド**（`stage`, `message`, `errKind` 等）を引用する。

- ある await の **直後のログが出ている** → その手前までは成功
- **`catch` にだけ来て `stage` が Y** → Y の途中の await（またはその同期処理）が疑わしい
- エラーが **握りつぶされて別メッセージ** になっている場合、**原因箇所に追加のログ** を入れて再実行する

---

## ステップ6: 修正する

- **ログで CONFIRMED になった原因** にだけ最小差分で直す。
- 仮説が REJECTED になったときに入れた **推測ベースのガード** は削除し、**証拠のある変更だけ** を残す（デバッグ汚染の防止）。

---

## ステップ7: 修正後の検証（ログは残す）

- 計測は **まだ削除しない**。
- **再検証の前に**、既存のログファイルが残っていれば **削除してからクリーンな状態で再検証** する（pre-fix のログが混在するのを防ぐ）。
- `runId: post-fix` などで前後比較しやすくする。
- 期待するログ（成功パスのみ、 `catch` が出ない等）を文章で定義してから、ユーザーに再現してもらう。

---

## ステップ8: 計測の掃除

ユーザーが **問題解消を確認**したら、`fetch` / 追記ログ / `stage` 専用変数など **デバッグ専用コードを削除** する。**本番の修正ロジックは残す**。

### ログファイルの掃除

デバッグ中に生成した **NDJSON ログファイル**（ステップ3 で追記先に指定したファイル）も削除する。**自分のセッション用ログファイルだけ** を対象とし、他セッションのファイルには触れない。

---

## 再利用用の短いプロンプト（貼り付け用）

次をそのまま指示に近い形で渡せる。

```text
ランタイム証拠でデバッグして。
1) 失敗経路に対して仮説を3〜5個立て、IDを付ける。
2) ステージ変数と最小限のログ（hypothesisId付き）を入れる。秘密は出さない。
3) 再現前に指定ログファイルだけクリアする。
4) 再現手順を書いてユーザーに実行してもらう。
5) ログで各仮説を CONFIRMED/REJECTED し、根拠を引用する。
6) 証拠がある原因だけ最小修正する。rejectした仮説のコードは戻す。
7) 計測を残したまま post-fix で再検証し、問題なければ計測を削除する。
推測だけでの修正と、sleep による回避は禁止。
```
