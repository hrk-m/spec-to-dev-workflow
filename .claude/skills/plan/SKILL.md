# /plan — 詳細設計コマンド

ユーザーと対話しながら「何を・なぜ・どう作るか」を決め切り、**詳細設計レベルの処理フロー**を確定する。機能の既存仕様は `plans/` を主に参照し、関連する `specs/` が存在する場合だけ補助参照する。

開始条件・停止条件・戻り先・完了条件の正本は `AGENTS.md` と `.claude/rules/workflow.md` とし、このスキルはそこから逸脱しない。

## いつ使うか

| タイミング | 説明 |
| --- | --- |
| 新機能の開発を始める前 | 何を作るか・なぜ作るかをユーザーと合意するために使う |
| PRD 未整備の純粋バグ修正に着手する前 | バグ箇所・再現条件・期待挙動を明確化して `plans/` に落とし込むために使う |
| 既存機能を拡張・変更する前 | 変更範囲と処理フローをユーザーと確認するために使う |
| `plan-writer` を実行する前 | 要件書き出しに必要な目的・ゴール・フローを確定するために使う |
| `/arch-refactor` や `/e2e-gen` から要件・設計差分で戻ったとき | 合意済み要件を更新するために使う |

**使ってはいけないタイミング**:
- すでに目的・ゴール・フローが合意済みの場合（重複になる）
- 設計変更を伴わない純粋なバグ修正・リファクタリングで、対象機能の PRD が既存かつ必要な計画が既に揃っている場合（ただし、ショートカット開始でも最終ゲート `/e2e-gen → /impl-done` はスキップ不可）

> **PRD 未整備の純粋バグ修正**: `/plan` から開始し、バグ箇所・再現条件・期待挙動を明確化して `plans/` に落とし込んだ後は、通常の修正サイクル（`/plan-writer` 以降）に進む。`/impl` へのショートカットは不可。

## 実行フロー概要

```
スキル起動
  ↓
実装パターンの把握（自動）
  - `sample-api-agent` に技術スタック・アーキテクチャの調査を依頼する
  - `sample-front-agent` に技術スタック・アーキテクチャの調査を依頼する
  ↓
影響 plans の確認（AskUserQuestion multiSelect）
  - `Glob` で `plans/*/*/prd.md` と `plans/shared/*.md` を収集し、既存機能一覧を取得する
  - `AskUserQuestion(multiSelect: true)` で今回の計画に関係する機能を選択させる
  - 選択された `plans/{ドメイン名}/{verb-noun}/prd.md` または `plans/shared/{処理名}.md` を `Read` して仕様を把握する
  - `specs/` は存在する場合のみ補助参照する
  - PRD 未整備の純粋バグ修正では、不具合箇所・再現条件・期待挙動をここで明確化する
  - 「新機能を追加する」が含まれる場合は `AskUserQuestion` で機能名を確認する
  ↓
Phase 1: 目的・ゴールの合意（ユーザーと対話）
  ↓ 合意が取れたら(`AskUserQuestion` でユーザーの最終合意をとる)
Phase 2: 詳細処理フローの設計（ユーザーと対話）
  ↓ 合意が取れたら(`AskUserQuestion` でユーザーの最終合意をとる)
完了報告 → plan-writer スキルへ案内
```

## 完了条件

- 対象機能と「なぜ作るか・完成形」がユーザーと合意されている
- エンドポイントごとに 5-1〜5-5 を 1 ステップずつ確認し、各エンドポイントの最終合意まで完了している
- 全エンドポイントの合意済み要件をチャットにまとめて提示している
- 次のステップとして `/plan-writer` を案内している

## 実行手順（この順序で必ず行う）

### 実装パターンの把握

`sample-api-agent` と `sample-front-agent` を **並列** で起動し、各ディレクトリの技術スタック・アーキテクチャを調査させる。
コードの参照はこれらのエージェントに委譲し、Phase 1・2 のフロー設計に必要な実装上の前提を把握する。

---

### 影響 plans の確認

1. `Glob` で `plans/*/*/prd.md` と `plans/shared/*.md` を取得し、既存の機能一覧を収集する
2. `AskUserQuestion(multiSelect: true)` で今回の計画に影響する機能を確認する
   - options に既存機能名を列挙し、末尾に **「新機能を追加する」** を必ず含める
   - 例: `["user-management", "hello", ..., "新機能を追加する"]`
   - 選択された機能の `plans/{ドメイン名}/{verb-noun}/prd.md` または `plans/shared/{処理名}.md` を `Read` して仕様を把握する
   - `specs/` は存在する場合のみ補助参照する
   - PRD 未整備の純粋バグ修正でも `/plan` から開始し、不具合箇所・再現条件・期待挙動をこの段階で明確化する
   - 「新機能を追加する」が含まれる場合: `AskUserQuestion` で機能名を確認する

---

### Phase 1 — 目的・ゴールの合意

> **用途**: 対象機能と「なぜ作るか・完成形」をユーザーと合意する。合意なしに Phase 2 へ進めない。

@references/phase1-goal-alignment.md

### Phase 2 — 詳細処理フローの設計

> **用途**: 合意した目的・ゴールをもとに、エンドポイント・処理ステップ・エラーケースを確定する。ユーザーの合意なしに完了とみなさない。
> **実装パターン参照の目的**: API・フロントエンドの技術スタックとアーキテクチャ上の前提を把握するために使用する。
> 命名規則・ディレクトリ構造・変数名・関数名・レイヤー区別などの具体的な実装詳細は Phase 2 の処理フローに含めない。

@references/phase2-flow-design.md

## 制約

- ショートカットで `/impl` から開始した場合も、最終ゲート（`/e2e-gen → /impl-done`）はスキップ不可
- **Phase 2 確認手順（最優先・絶対厳守）**: Phase 2 に入ったら `phase2-flow-design.md` のステップ 5-1〜5-5 を必ず **1 ステップずつ** `AskUserQuestion` で確認する。全体フローを先に提示することは禁止。完了条件に「フローが整理されている」とあっても、この確認手順を経てから整理すること。
- 不明点は推測で埋めない。必ず `AskUserQuestion` で確認する
- **用語統一**: 同じ概念には同じ用語を使う。独自の呼び方が必要な場合は定義を明記する
- 機能の既存仕様は `plans/` を主に参照し、関連する `specs/` が存在する場合だけ補助参照する

## フォールバック

| 状況 | 対応 |
| --- | --- |
| `plans/` に機能が存在しない | options に「新機能を追加する」のみ表示して選択させる |
| 引数が空 | `AskUserQuestion` で対象機能の選択から開始 |
| 目的・ゴールが曖昧 | `AskUserQuestion` で深掘りして合意が取れるまで確定しない |
| 処理ステップに不明点がある | `AskUserQuestion` で確認が取れるまで次フェーズに進めない |
| 回答が空・未選択 | 推測で進めず質問を再提示する |

---

## ツール案内

- **sample-api-agent / sample-front-agent**: スキル起動直後に各ディレクトリの実装パターン（技術スタック・アーキテクチャ）を把握するために使う。コードの参照はこれらのエージェントに委譲する
- **Glob / Read**: 影響 plans の確認ステップで `plans/` の既存機能一覧・仕様を収集するために使う。関連する `specs/` が存在する場合は補助参照する
- **AskUserQuestion**: Phase 1・2 の各合意ポイントで使う
  - 選択肢のどれも該当しない場合、ユーザーは自動的に表示される **"Other"** を選んで自由テキストを入力できる
  - `options` に "Other" を手動追加する必要はない（ツールが自動付与する）
  - ユーザーが "Other" を選択した場合は、入力内容を要件に反映してから次フェーズへ進む
