# Phase 1 — 目的・ゴールの合意

> 影響 plans の確認（Glob + AskUserQuestion multiSelect）は SKILL.md の「影響 plans の確認」ステップで完了済み。

1. ユーザーの入力と選択された plans の仕様から **目的**（なぜ作るか）と **ゴール**（完成した状態）を引き出す
   - 不明な場合は推測せず `AskUserQuestion` で確認する
   - PRD 未整備の純粋バグ修正では、ここで不具合箇所・再現条件・期待挙動を明確化し、目的・ゴールに含める
2. 目的・ゴールをチャットに整理して提示し、`AskUserQuestion` でユーザーの最終合意を得る
   - 合意が取れるまで Phase 2 に進めない

3. バックエンド API の新規作成・変更が含まれる場合は、以下の 3 観点をユーザーと合意する

   | 観点 | 確認内容 |
   |---|---|
   | **返却データ量** | 1 レスポンスに含まれるデータ件数・サイズの見積もり |
   | **処理の重さ** | DB クエリ数・外部 API 呼び出し・計算量の見積もり |
   | **アクセス頻度** | 想定リクエスト数（QPS）・ピーク時の同時接続数の見積もり |

   これらをもとに「単一 API で賄うか・API を分離するか」もゴールの一部として合意する。
   以下の目安を参考に判断する：

   | 観点 | 分離を検討するサイン |
   |---|---|
   | 返却データ量が大きい | 一覧系で 100 件超、レスポンスが 1MB 超 → ページネーション / サマリー + 詳細 API 分離 |
   | 処理が重い | 複数テーブル JOIN / 外部 API 複数呼び出し / バッチ処理相当 → 非同期処理・ジョブ分離 |
   | アクセス頻度が高い | 読み取りが高頻度・書き込みが低頻度で混在 → 読み書き分離（CQRS）/ キャッシュ戦略 |
   | 分散トランザクションが必要 | 複数サービス跨ぎ / 補償ロジックが必要 → Saga パターン |
   | 境界が異なるドメインが混在 | 責務が異なる集合体が 1 API に混在 → Bounded Context で分割 |
