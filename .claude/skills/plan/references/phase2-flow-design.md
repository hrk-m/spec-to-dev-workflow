# Phase 2 — 詳細処理フローの設計

5. **Phase 2 に入ったら最初に必ず** エンドポイントごとに以下の 4 ステップを **1 ステップずつ順番に** `AskUserQuestion` で確認する。

   > - 5-1から5-4のステップは一つづつ`AskUserQuestion` で確認すること。

   > **禁止事項（厳守）**:
   > - 複数ステップをまとめて一度に提示すること
   > - 前のステップの合意が取れる前に次のステップに進むこと
   > - この確認を省略・後回しにして処理フロー全体を先に提示すること
   > - 複数エンドポイントをまとめて確認すること（必ず 1 エンドポイントずつ行う）

   ---

   **確認ステップ 5-1: リクエスト・バリデーション**（対象エンドポイント名を明示）

   - 空か否か（null, undefined, "", [], {}）
   - 0文字・0件許可するか否か
   - 最小・最大値（数値・文字列長）
   - 正数・負数・0制限
   - 小数桁数制限
   - 指定文字種のみ（英数字など）
   - 特定記号・制御文字禁止
   - メール、URL、電話番号、日付などの形式チェック
   - パスワード：長さ・大小文字・数字・記号・二重入力チェック
   - リストの要素型・範囲・重複チェッ
   - オブジェクトの必須キー・型・不要キー禁止
   - DBや外部IDの存在・一意性チェック
   - AならB必須など依存チェック
   - クライアント側：リアルタイムチェック・送信前チェック・エラーメッセージ表示制御

   ````
   ## 処理フロー: {機能名} — {操作名}

   ### エンドポイント
   - メソッド: POST / GET / PUT / DELETE
   - パス: /api/v1/xxx
   - 認証: 必要 / 不要

   ### リクエスト
   | フィールド | 型 | 必須 | 説明 |
   |---|---|---|---|
   | field_name | string | ✓ | ... |
   ````

   → `AskUserQuestion` で「このまま進む / 修正がある」を確認する。
   → 合意が取れてから **5-2 へ進む**。

   ---

   **確認ステップ 5-2: 処理ロジック（追加・更新）**（対象エンドポイント名を明示）

   フロントエンドとバックエンドの処理フローを以下の形式でチャットに提示する。

   ````
   ### フロントエンド 処理フロー

   ```
   1. 開始
   2. {操作名} ボタン押下 / ページ読み込み
   3. {Component} → {APIクライアント関数}: {METHOD} {パス} 送信
   4. レスポンスは成功？
      - Yes（200）→
         5. state 更新
         6. 画面に反映
         7. 終了
      - No（4xx・5xx）→
         5. エラーメッセージ表示
         6. 終了
   ```

   ---

   ### バックエンド 処理フロー

   ```
   1. 開始
   2. クライアントから HTTP リクエスト（`{METHOD} {パス}`）を受信
   3. 入力バリデーションを実施
      - `{フィールド名}` が空でないこと
      - `{その他条件}` を満たしていること
   4. バリデーション通過？
      - No →
         - 400 Bad Request `{ "error": "VALIDATION_ERROR" }` を返す
         - 終了
      - Yes →
         5. ビジネスロジックを実行（リクエスト内容に基づいた処理）
         6. データアクセスを実行（DB やストレージに対して操作）
         7. DB への `{SQL操作}` を実行（対象テーブル: `{table_name}`）
         8. DB 操作成功？
            - No →
               - 500 Internal Server Error `{ "error": "INTERNAL_ERROR" }` を返す
               - 終了
            - Yes →
               9. DB の結果を、上位の処理層に返却
               10. 最終的にクライアントに {ステータスコード} + JSON を返す
               11. 終了
   ```
   ````

   → `AskUserQuestion` で「このまま進む / 修正がある」を確認する。
   → 合意が取れてから **5-3 へ進む**。

   ---

   **確認ステップ 5-3: DB 操作（DB 変更がある場合のみ）**（対象エンドポイント名を明示）

   DB 変更がない場合はこのステップを **スキップ** し、即座に 5-4 へ進む。
   DB 変更がある場合は対象テーブル・SQL 操作（INSERT / UPDATE / SELECT など）をチャットに提示する。
   → `AskUserQuestion` で「このまま進む / 修正がある」を確認する。
   → 合意が取れてから **5-4 へ進む**。

   ---

   **確認ステップ 5-4: レスポンス・エラーケース**（対象エンドポイント名を明示）

   正常系レスポンスのフィールドと型、エラーケース一覧（全パターン・発生レイヤー付き）を以下の形式でチャットに提示する。

   ````
   ### レスポンス（正常系）
   - ステータス: 200 OK
   ```json
   { "field": "value" }
   ```

   ### エラーケース一覧
   ```
   - バリデーション失敗 → 400 Bad Request `{ "error": "VALIDATION_ERROR" }`
   - 未認証 → 401 Unauthorized `{ "error": "UNAUTHORIZED" }`
   - リソース未存在 → 404 Not Found `{ "error": "NOT_FOUND" }`
   - 想定外エラー → 500 Internal Server Error `{ "error": "INTERNAL_ERROR" }`
   ```
   ````

   → `AskUserQuestion` で「このまま進む / 修正がある」を確認する。
   → 合意が取れてから **次のエンドポイントの確認、またはすべて完了なら手順 6 へ進む**。

6. ステップ 5-1〜5-4 の合意内容をすべて反映した詳細処理フローをチャットに提示し、最終合意を取る（合意が取れるまで繰り返す）

7. 複数エンドポイントがある場合は **エンドポイントごとに手順 5〜6 を繰り返す**

8. フロー確定後にタスク名を英小文字 kebab-case で決定する
   - 例: `ユーザー管理機能` → `user-management`

## フロー図設計の注意事項

- **FE・BE 両方必須**: フロントエンドとバックエンドの処理フローは **常に両方** 提示する。片方のみ・省略は禁止
- **表示形式**: 処理フローは **必ず番号付きステップ形式**（コードブロック内に `1. 開始` `2. ...` の形式）で記述する。テーブル形式での表示は禁止
- **条件分岐の記述**: 分岐は `- Yes →` / `- No →` でインデントして表現する（`├─` / `└─` の記号は使わない）
- **用語統一**: 同じ概念には同じ用語を使う（例: 「ユーザー」と「利用者」を混在させない）
- **造語回避**: 既存の技術用語を使い、独自の呼び方が必要な場合は定義を明記する
- **簡潔な記述**: 各ステップの説明は 30〜40 字程度を目安にする
