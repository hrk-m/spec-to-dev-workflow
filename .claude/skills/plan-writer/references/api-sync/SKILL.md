# /api-sync — 現コードから plans/prd.md を生成・更新する

## 役割

`sample-api/` の現コードを読み取り、機能固有の `plans/{ドメイン名}/{verb-noun}/prd.md` を生成または更新する。
**通常フローでは `/e2e-gen` の内部手順として実行する。例外ルートとして `/spec-update → plans-sync` 経由でも実行できる。機能固有の `plans/{ドメイン名}/{verb-noun}/prd.md` が実装対象に含まれる場合のみ起動し、shared-only タスクでは起動しない。**

- **新規作成**: ファイルが存在しない場合、テンプレートをもとに生成する
- **更新**: ファイルが存在する場合、コードと乖離しているセクションのみ更新する
- **ドリフト検出**: `specs/` の `## 確認観点` と prd.md の `確認ステップ 5-5: ユニットテストケース` を比較し、差異を報告する

---

## 用語

| 用語 | 意味 | 例 |
|---|---|---|
| `{ドメイン名}` | `plans/` 配下のドメインディレクトリ名（単数形） | `group` |
| `{verb-noun}` | API 操作単位の名前 | `list-groups` |

---

## 実行フロー

### Step 1: アーキテクチャ構造を把握する

orchestrator が直接 Glob / Grep / Read で調べ、レイヤー → ディレクトリのマッピングを確定する。subagent は起動しない。

**sample-api**
- 入力受付（ルーティング・ハンドラ）のディレクトリパス
- 業務処理（サービス）のディレクトリパス
- データ処理（リポジトリ）のディレクトリパス
- 共通定義（ドメインモデル）のディレクトリパス
- マイグレーションのディレクトリパス

**sample-front**
- 画面（ページ・コンポーネント）のディレクトリパス
- API 呼び出しのディレクトリパス
- 型定義のディレクトリパス

---

### Step 2: テンプレートを読み込む

`.claude/skills/plan-writer/references/api-sync/templates/prd.md` を `Read` して構造を把握する。
見出し名・順序・記述ルールに差がある場合は、テンプレートを正として合わせる。

---

### Step 3: エンドポイント一覧を収集する

Step 1 で把握した入力受付側のディレクトリから登録済みエンドポイントを収集する。

```bash
grep -rn "\.GET(\|\.POST(\|\.PUT(\|\.PATCH(\|\.DELETE(" {入力受付側ディレクトリ}/
```

---

### Step 4: `{ドメイン名}` / `{verb-noun}` を自動決定する

ユーザー確認不要。以下のルールで自動決定する。

**`{ドメイン名}`**: `plans/` 配下のディレクトリ名として、パスのリソース名を単数形にしたものを使う（例: `/api/v1/groups` → `group`）

**`{verb-noun}`**:

| HTTP メソッド | パスの特徴 | verb-noun |
|---|---|---|
| `GET` | コレクション（`/groups`） | `list-{リソース}` |
| `GET` | 単体（`/groups/:id`） | `get-{リソース単数形}` |
| `POST` | コレクション | `create-{リソース単数形}` |
| `PUT` / `PATCH` | 単体 | `update-{リソース単数形}` |
| `DELETE` | 単体 | `delete-{リソース単数形}` |
| `GET` | サブリソース・コレクション（`/groups/:id/members`） | `list-{親単数形}-{子}` |
| `POST` | サブリソース・コレクション（`/groups/:id/members`） | `add-{親単数形}-{子}` |
| `DELETE` | サブリソース・コレクション（`/groups/:id/members`） | `delete-{親単数形}-{子}` |
| `GET` | 修飾サブリソース（`/groups/:id/non-members`） | `list-{修飾語}-{親単数形}-{子}` |

例: `GET /api/v1/groups` → ドメイン名 `group` / verb-noun `list-groups` / 出力先 `plans/group/list-groups/prd.md`
例: `POST /api/v1/groups/:id/members` → ドメイン名 `group` / verb-noun `add-group-members` / 出力先 `plans/group/add-group-members/prd.md`
例: `GET /api/v1/groups/:id/non-members` → ドメイン名 `group` / verb-noun `list-non-group-members` / 出力先 `plans/group/list-non-group-members/prd.md`

---

### Step 4.5: エンドポイントをドメイン単位でグループ化する

Step 3・4 の結果から、エンドポイントを `{ドメイン名}` 単位でグループ化する。

例:
```
group: list-groups / get-group / create-group / update-group / delete-group
       list-group-members / add-group-members / delete-group-members / list-non-group-members
user:  list-users
auth:  get-me
```

このグループ単位が Step 5〜7 の並列実行単位となる。

---

### Step 5: ドメイン単位で並列起動してコードを読み PRD を生成する

既存 prd.md の有無を確認する場合は `plans/**/prd.md` パターンのみ使用する（`.ralph-alignment.md` / `.ralph-impl.md` 等を含む `plans/**/*.md` は使用しない）。

**ファイルの有無によって処理を分岐する。**

- **新規作成（ファイルなし）**: Step 6 の全セクション生成 → Step 7 の `Write` で書き出す
- **既存ファイルの更新（ファイルあり）**: Step 6 をスキップし、Step 7 のファクト照合のみを実行する

**ドメインごとにエージェントを並列起動する。** 各エージェントは担当ドメインのファイルを一度読み、配下の全エンドポイントを順次処理する。

```
並列
├── {domain}-agent（例: group）
│   ├── ① 担当ドメインの共有ファイルを読む
│   ├── ② FE の担当コードを読む
│   └── ③ 配下のエンドポイントを順次処理（新規: Step 6→7 / 既存: Step 7 のみ）
│       ├── list-groups → prd.md 生成・書き出し
│       ├── create-group → prd.md 生成・書き出し
│       └── ...
├── {domain}-agent（例: user）
│   └── 同上
└── {domain}-agent（例: auth）
    └── 同上
```

**同じ役割領域内のエンドポイントはファイルを共有するため、重複読み取りを避けるために必ず順次処理すること。**

---

### Step 6: prd.md の各セクションをコードから生成する

Step 2 で読んだテンプレートの各セクションを、Step 5 で読んだコードから埋める。
概要・5-1・5-4・5-5・要件・対象外はテンプレートのサンプル行を参考にコードから埋める。
以下では特に判断が難しい 5-2・5-2-FE・5-3 の記述ルールを示す。

#### 5-2 バックエンド処理フローの記述ルール

**原則: テンプレートの `確認ステップ 5-2: バックエンド処理フロー` と `確認ステップ 5-4: レスポンス・エラーケース` に合わせ、コードに存在する処理を 1 つも省略しない。**

- 同目的の連続処理は 1 番号にまとめる
- 同一フィールドの取得・検証は 1 番号にまとめる
- 粒度は「索引」レベル
- エラー返却は抽象化する
- 低レベル実装詳細は書かない
- 関数名・メソッド名・内部の実装名は書かない
- 「〜など」「〜等」でまとめない
- テストファイルで不足分を補う

**記述例（正しい粒度）:**

| ❌ 書いてはいけない | ✅ 書くべき |
|---|---|
| `validateGroup(ctx, name)` を呼び出す | グループ名の長さと文字種を確認する |
| `SELECT * FROM groups WHERE deleted_at IS NULL` を実行する | 削除済みでないグループ一覧と総件数を取得する |
| `limit` を 20 にデフォルト設定する | 件数上限パラメータを取得し、未指定の場合はデフォルト値を使用する |
| DB エラーが返ってきた場合など | DB エラーの場合 → 500 Internal Server Error 終了 |
| `group.SetUpdatedAt(time.Now())` を実行する | 更新日時を現在時刻に設定する |

以下の項目を全て記載すること（該当しない項目のみ省略可）。

- **入力バリデーション**: テンプレート 5-1 に合わせ、同一フィールドは 1 つにまとめて、何を確認・検証するかを日本語で書く。エラー分岐末尾に「終了」
- **ビジネスロジック**: テンプレート 5-2 に合わせ、入力処理とメイン DB 操作（INSERT・UPDATE・DELETE・主要 SELECT）の間に位置する処理（事前存在確認・権限確認・パラメータ変換・デフォルト値設定）を実行順に書く
- **データアクセス**: テンプレート 5-2 に合わせ、何を確認・取得・更新するかを書く
- **エラーマッピング**: テンプレート 5-4 に合わせ、`{条件} → {HTTP ステータス}` で書く。各エラー分岐末尾に「終了」
- **正常レスポンス**: テンプレート 5-2 / 5-4 に合わせ、ステータスコードと返却内容を書く。末尾に「終了」
- **凡例**: ブロック直前に 1 行で書く

#### 5-2 自己検証（生成直後に必ず実行）

5-2 を書き終えた後、対象ファイルを再度参照し、以下を 1 対 1 で照合する。漏れがあれば即座に追記する（再ループ不要）。

| 照合対象 | 確認内容 |
|---|---|
| 入力バリデーションの全 `if` 分岐 | 5-2 に対応する行があるか |
| ビジネスロジックの全 `if` 分岐 | 5-2 に対応する行があるか |
| データアクセスの全 `if` 分岐 | 5-2 に対応する行があるか |
| `switch` / `case` の全ケース | 5-2 に対応する行があるか |
| 入力バリデーション | 何を取得・確認・検証するかが日本語で説明されているか |
| データアクセス | SQL 構文ではなく操作の目的が日本語で説明されているか |
| 各エラー分岐末尾 | 「終了」が記載されているか |
| 正常レスポンス末尾 | 「終了」が記載されているか |
| 全エラーケース | HTTP ステータスへのマッピングが記載されているか |
| 凡例 | フローブロック直前に `→` と HTTP ステータスの意味が日本語で記載されているか |

---

#### 5-2-FE フロントエンド処理フローの記述ルール

**原則: API 呼び出しを起点に、UI トリガー → リクエスト → レスポンス処理 → UI 更新 を書く。**

`sample-front/` の該当コードを読み、処理を省略なく記載する。

以下の項目を全て記載すること（該当しない項目のみ省略可）。

- **コンポーネント・フック名**: 呼び出し元の名前を書く
- **正確な API URL**: パラメータ込みの完全な URL を書く
- **レスポンス分岐**: Yes（成功）/ No（失敗）で分ける。各分岐末尾に「終了」
- **状態管理**: 状態ごとに目的を一文で書く（例: 一覧データを offset 単位でキャッシュする / ローディング中かどうかを管理する / エラーメッセージを保持する）。変数名は書かない
- **ローディング状態**: 表示内容を書く
- **無限スクロール**: ある場合は条件と追加方法を書く
- **デバウンス**: ある場合は待機時間とリセット条件を書く
- **成功後の副作用**: データ再取得・画面遷移・フォームリセット・モーダルクローズ・親コンポーネントへの再取得依頼のうち、実装されているものを書く
- **凡例**: ブロック直前に 1 行で書く

#### 5-2-FE 自己検証（生成直後に必ず実行）

5-2-FE を書き終えた後、対象ファイルを再度参照し、以下を 1 対 1 で照合する。漏れがあれば即座に追記する（再ループ不要）。

| 照合対象 | 確認内容 |
|---|---|
| コンポーネント名・フック名 | 5-2-FE に明記されているか |
| 正確な API URL（パラメータ込み） | 5-2-FE に記載されているか |
| レスポンスの全分岐（Yes/No 形式） | 5-2-FE に対応する行があるか |
| 状態管理 | 変数名ではなく役割が日本語で説明されているか |
| 無限スクロール | 実装されている場合に追加読み込み条件が日本語で説明されているか |
| デバウンス | 実装されている場合に待機時間とリセット条件が日本語で説明されているか |
| 各分岐末尾 | 「終了」が記載されているか |
| 凡例 | フローブロック直前に `→` の定義が記載されているか |

#### 5-3 ファイル配置

**原則: 関与した全ファイルを列挙し、役割は具体的に書く。**

---

### Step 7: 書き出す

書き出し先は Step 4 のマッピングに従い `plans/{ドメイン名}/{verb-noun}/prd.md` に固定する。
`plans/shared/{処理名}.md` はこのスキルの対象外とし、必要な場合は `plan-writer` 側で直接書き出す。

#### 新規作成（ファイルが存在しない場合）

`Write` でテンプレートに従い全セクションを生成する。

#### 既存ファイルの更新（ファイルが存在する場合）

**べき等原則**: 同じコードから 2 回連続で実行しても、2 回目は変更ゼロになること。

更新判断は「ファクトの照合」で行う。セクション全体を生成し直して比較するのではなく、以下のファクトをコードから抽出し、既存 prd.md の対応箇所に正確に含まれているかを確認する。含まれていない場合のみ `Edit` で最小差分を加える。手動追記は保持する。

| セクション | 照合するファクト | Edit してよい条件 |
|---|---|---|
| 概要テーブル | HTTP メソッド・パス・認証要否 | ファクトが実装と異なる場合のみ |
| 5-1 バリデーション一覧 | フィールド名・型・必須/任意・エラー HTTP ステータス | 行の追加・削除・値の変更がある場合のみ |
| 5-2 BE 処理フロー | 処理ステップの増減・エラー HTTP ステータスの変更 | ステップが追加/削除された、またはステータスコードが変わった場合のみ |
| 5-2-FE FE 処理フロー | コンポーネント名・フック名・API URL・レスポンス分岐の増減 | これらが変わった場合のみ |
| 5-3 ファイル配置 | ファイルパスの存在 | パスが実在しない、または新規ファイルが追加された場合のみ |
| 5-4 エラーケース一覧 | エラー条件・HTTP ステータス | 行の追加・削除・ステータス変更がある場合のみ |
| 5-5 テストケース | 実際のテストファイルに存在する関数名・観点 | 実在しない関数が記載されている、または実在する関数が未記載の場合のみ |

**絶対に触ってはいけないもの（ファクトに変化がなければ編集しない）**:
- 凡例テキスト（`→ は…`）の言い回し
- 既存の処理ステップの日本語表現
- 既存のコメント・手動追記
- インデント・空行・コードブロックの整形

**5-2・5-2-FE に Edit または追記が必要な場合は、Step 6 の記述ルールと自己検証チェックリストに従うこと。**

**5-2-FE セクションの削除**: `sample-front/` に該当する FE 実装が存在しないことが確認できた場合に限り、5-2-FE セクションをまるごと削除する。

### Step 8: specs とのドリフトを検出する
