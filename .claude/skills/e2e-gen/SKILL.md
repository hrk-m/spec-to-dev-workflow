# /e2e-gen — E2E テスト生成スキル

`/arch-refactor` が REVIEWED を出力した後にだけ実行する。
このスキルの責務は、`plans/` の最終同期と、E2E テストの追加・検証である。

## 前提

- `/arch-refactor` が `REVIEWED` を出力している
- 修正サイクルを抜けている
- `/impl-done` はまだ実行していない

## 実行手順

### Step 1: `plans/` を最終同期する

以下のエージェントを並列起動し、両方の完了を待ってから Step 2 へ進む。

**api-sync エージェント**（常に起動・必須）

> `.claude/skills/plan-writer/references/api-sync/SKILL.md` を Read し、記載されたフロー（Step 1〜Step 9）に従って api-sync を実行してください。
> 実装コードの内容と対象 PRD の内容を照合し、差分がある箇所のみ `plans/` を更新してください。

**schema-sync エージェント**（DB 変更がある場合のみ起動）

> `.claude/skills/plan-writer/references/schema-sync/SKILL.md` を Read し、記載されたフロー（Step 1〜Step 11）に従って schema-sync を実行してください。
> `plans/schema.md` を必要に応じて更新してください。

- api-sync は `/e2e-gen` 起動時に必ず実行し、PRD を実装に合わせて最新化する
- schema-sync は DB 変更がある場合のみ起動する
- つまり、`plans/` の最終同期と必要に応じた PRD / schema 更新は `/e2e-gen` の責務とする

### Step 1 の停止条件

- 最低要件 / API 契約 / エラー挙動 / UX フローの解釈が揺れる差分が見つかった場合は問題内容をユーザーに報告してスキルを停止する（`/plan` には戻らない）
- 実装方針 / 追加ファイル / テスト方針の差分が見つかった場合は、仕様・設計の問題なら問題内容をユーザーに報告してスキルを停止する（`/plan` には戻らない）。実装のみの問題なら内容を蓄積し、スキル終了時にまとめてユーザーに報告し `/impl` の実行を促す（このスキル自身は実装しない）
- 停止条件①・停止条件②の仕様・設計問題のいずれにも該当しない場合は次のステップへ進む（停止条件②の実装のみ問題はバグを蓄積しつつ次のステップへ進む）

### Step 2: 既存状況を把握する

以下を並列で確認する。

1. 同期後の機能固有 `prd.md`
   - 機能固有 PRD がない共通処理のみの変更では `plans/shared/{処理名}.md` を正本として読む
   - `plans/shared/*.md` があれば補助参照も含めてテスト対象を洗い出す
2. `e2e/tests/` の既存テスト
3. `e2e/entrypoint.sh` のシードデータ

### Step 3: 10 カテゴリ観点を一括確認する

`git diff` で `plans/` の変更差分を取り、同期後の PRD / `plans/shared/*.md` / 既存 E2E / seed データと照合して No.1〜No.10 の全カテゴリを一括分析する。

**分析・提示手順**

1. 全カテゴリを横断して以下の 2 区分で観点を洗い出し、一覧でまとめて提示する

   | 区分 | 意味 |
   | --- | --- |
   | ✅ 追加推奨 | 今回の変更で新たにカバーすべき観点 |
   | 🗑️ 削除推奨 | 仕様変更・削除により不要になった既存テスト観点 |

   - `No.{カテゴリ番号} {カテゴリ名}` でグルーピングして提示する
   - 追加・削除のいずれも導出できないカテゴリは省略し、省略カテゴリを末尾に一覧する
   - 変更不要な既存テストは `変更なし: No.X, No.Y, ...` の 1 行サマリーにとどめる

2. `AskUserQuestion` で `OK / 修正 / コメント` を一括確認する
   - **OK** — 提示内容をそのまま採用して Step 4 へ進む
   - **修正** — ユーザーのコメントに従って修正し、再提示して再確認する
   - **コメント** — コメントを反映して再提示して再確認する

確認が取れたら Step 4 に進む。

**カテゴリ定義（分析の参照用）**

| No. | カテゴリ | 確認内容 |
| --- | --- | --- |
| 1 | ユーザー視点の基本フロー | 主操作の流れが通るか |
| 2 | ビジネスコアのシナリオ | 価値が直接出る主要フローか |
| 3 | 入力・データ・バリデーション | 正常入力と代表的な入力エラー |
| 4 | API・外部連携の整合性 | フロント、バック、DB、外部 API の整合 |
| 5 | 画面遷移・UI 挙動 | 遷移、ローディング、完了、失敗時の見え方 |
| 6 | リグレッション防止 | 壊したくないメインユースケース |
| 7 | ブラウザ・環境差分 | 実行環境差分の考慮 |
| 8 | 権限・認可・セキュリティ | 権限制御や保護すべき導線 |
| 9 | 障害・例外系 | タイムアウト、通信失敗、外部依存失敗 |
| 10 | データ準備・テスト前提 | seed(データが足りなければ追加)、fixture、前提データの妥当性 |

### Step 4: テストケースリストを確定する

Step 3 でカテゴリごとに確定した観点を、既存カバー済み / 新規追加に分けて一覧化し、`AskUserQuestion` で最終確認する。

### Step 5: `ralph-loop` で E2E を実装する

1. `e2e/.ralph-e2e-gen.md` を作成または更新する
2. `ralph-loop` を起動する

   ```
   Read and follow e2e/.ralph-e2e-gen.md --completion-promise "E2E COMPLETE" --max-iterations 5
   ```

3. `sample-front-agent` に `e2e/tests/` の実装を委譲し、`make e2e` を実行させる
   - frontend 変更の有無にかかわらず、`e2e/tests/` の担当は `sample-front-agent`
4. 全 pass → `E2E COMPLETE` を出力してループ終了

## 戻り先ルール

- E2E テスト実装自体の問題なら `sample-front-agent` に修正依頼してイテレーション終了（外部に戻らない）
- BE / FE の実装バグが原因の場合はループを停止し、バグ内容を蓄積する。スキル終了時に全バグをまとめてユーザーに報告し `/impl` の実行を促す（このスキル自身は実装しない）
- 仕様・設計の問題ならループを停止し、問題内容をユーザーに報告してスキルを停止する（`/plan` には戻らない）
- `E2E COMPLETE` を出力したら `/impl-done` に進む

## バグ報告フォーマット

BE / FE の実装バグが検出された場合、スキルの最後に以下の形式でまとめて報告する。

```
## E2E バグ報告

以下の実装バグが検出されました。

| # | テスト名 | バグ内容 | 該当箇所（推定） |
|---|---|---|---|
| 1 | {テスト名} | {エラー内容} | {ファイル・コンポーネント名} |
| 2 | ... | ... | ... |

### 次のステップ

`/impl` で上記バグを修正し、`/arch-refactor` → `/e2e-gen` の順に再実行してください。
```

## 完了条件

- `plans/` の最終同期が完了している
- E2E ケースの最終合意が取れている
- `make e2e` が全 pass している
- `E2E COMPLETE` を出力している
